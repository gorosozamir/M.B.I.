1. Proceso de Requerimientos

Definición

El proceso de requerimientos es la actividad sistemática para identificar, analizar, documentar y gestionar las necesidades y expectativas de los stakeholders respecto a un sistema o software.

Fases principales

  • Elicitación: Recolección de requisitos mediante interacción con stakeholders
  • Análisis: Refinamiento y clarificación de requisitos
  • Especificación: Documentación formal de los requisitos
  • Validación: Verificación con stakeholders para asegurar que los requisitos reflejan sus necesidades
  • Gestión: Control de cambios y trazabilidad durante el ciclo de vida

Actores involucrados

  • Analistas de requisitos
  • Usuarios finales
  • Clientes
  • Desarrolladores
  • Gerentes de proyecto
  • Expertos en dominio
  • Otros stakeholders que influyen o usan el sistema

Ciclo de vida de los requerimientos

Incluye la identificación, análisis, documentación, validación, gestión de cambios y trazabilidad hasta la implementación y mantenimiento.

2. Técnicas de Obtención de Requisitos

Entrevistas

  • Conversaciones estructuradas con stakeholders
  • Obtener información cualitativa y detallada
  • Requiere preparación y selección adecuada de entrevistados
  • Uso de preguntas abiertas para profundizar

Cuestionarios

  • Permiten recopilar datos de un grupo amplio
  • Útiles para confirmar y validar información
  • Obtener opiniones de múltiples interesados

Observación

  • Estudio del entorno y actividades de los usuarios
  • Puede ser pasiva (sin interacción) o activa (con diálogo)
  • Útil para documentar procesos actuales
  • Detectar necesidades no expresadas

Talleres de trabajo (JAD/RAD)

  • Sesiones colaborativas entre usuarios expertos y analistas
  • Discutir, negociar y definir requisitos
  • Fomentan la cooperación y clarificación rápida

Análisis de documentos existentes

  • Revisión de documentación previa
  • Planes de negocio, contratos, especificaciones técnicas
  • Extraer requisitos históricos y actuales

Prototipos

  • Creación de modelos visuales o interactivos
  • Visualización del sistema por parte de usuarios
  • Retroalimentación temprana

3. Modelización y Especificación de Requisitos

Modelos funcionales

  • Casos de uso: Describen interacciones entre usuarios y sistema para capturar funcionalidades
  • Diagramas de flujo de datos: Representan el flujo de información y procesos dentro del sistema

Modelos estructurales

  • Diagramas de clases: Definen las entidades, atributos y relaciones
  • Diagramas entidad-relación (ER): Modelan la estructura de datos y sus relaciones

Modelos de comportamiento

  • Máquinas de estados: Representan estados y transiciones del sistema
  • Diagramas de secuencia: Muestran interacciones temporales entre objetos o componentes

Lenguajes formales vs. semi-formales

  • Formales: Usan notaciones matemáticas precisas para especificar requisitos (menos comunes en práctica)
  • Semi-formales: UML (Unified Modeling Language), BPMN (Business Process Model and Notation) y especificaciones textuales estructuradas, que facilitan la comunicación entre técnicos y usuarios

4. Tratamiento de Requisitos de Calidad y Seguridad

Requisitos no funcionales

  • Rendimiento
  • Usabilidad
  • Confiabilidad
  • Mantenibilidad
  • Seguridad

Métricas de calidad

  • Tiempos de respuesta
  • Tasa de errores
  • Disponibilidad

Requisitos de seguridad y privacidad

  • Controles para proteger confidencialidad, integridad y disponibilidad
  • Cumplimiento de normativas

Normativas y estándares

  • ISO/IEC 27001 para seguridad
  • OWASP para seguridad en aplicaciones web
  • ISO/IEC 25010 para calidad de software

5. Métodos Orientados a Objetivos

Consideran los requisitos como metas o intenciones que el sistema debe alcanzar.

Técnicas destacadas

  • KAOS: Modela objetivos, agentes y responsabilidades para derivar requisitos
  • i*: Enfocado en actores y sus dependencias
  • Goal-oriented Requirements Engineering (GORE): Facilita la identificación y análisis de objetivos para guiar el desarrollo

6. Especificación de Requisitos Temporales

Restricciones temporales

  • Definen condiciones relacionadas con el tiempo, como plazos o secuencias

Requisitos de rendimiento

  • Tiempos de respuesta
  • Throughput
  • Disponibilidad
  • Escalabilidad

Especificación con lógica temporal

Uso de notaciones formales como TCTL (Timed Computation Tree Logic) para expresar propiedades temporales y de tiempo real.

7. Verificación y Validación dentro del Proceso

Revisión de requisitos

  • Inspección sistemática
  • Detectar errores, inconsistencias o ambigüedades

Validación con stakeholders

  • Confirmar que los requisitos reflejan las necesidades reales
  • Entrevistas, talleres o prototipos

Técnicas de verificación

  • Inspecciones
  • Walkthroughs (recorridos guiados)
  • Pruebas de requisitos
  • Simulaciones

8. Asignación de Prioridades y Resolución de Conflictos

Criterios de priorización

  • MoSCoW: Must have, Should have, Could have, Won't have
  • Valor de negocio: Impacto en objetivos estratégicos
  • Riesgo: Complejidad o incertidumbre asociada

Resolución de conflictos

  • Negociación entre stakeholders para llegar a acuerdos
  • Análisis costo-beneficio
  • Compromiso

Técnicas:

  • Facilitación
  • Mediación
  • Análisis de impacto

9. Administración de Requisitos

Estándares de documentación

  • Uso de formatos claros y consistentes para describir requisitos

Trazabilidad

  • Relación entre requisitos, diseño, implementación y pruebas
  • Asegurar cobertura y gestión de cambios

Manejo de cambios y control de versiones

  • Procesos para evaluar, aprobar y registrar modificaciones

Herramientas de gestión

  • DOORS
  • Jira
  • Trello

10. Requisitos en Procesos Ágiles

Historias de usuario

  • Descripciones breves y simples de funcionalidades desde la perspectiva del usuario

Backlogs y épicas

  • Listas priorizadas de historias
  • Agrupaciones grandes de funcionalidades

Revisión y refinamiento continuo

  • Ajuste frecuente de requisitos con feedback constante

Colaboración constante con el cliente

  • Comunicación directa y continua para asegurar alineación con necesidades reales
